## üìã Documentation Standards

### File Naming Convention
- Use `UPPER_CASE_WITH_UNDERSCORES.md` for documentation files
- Platform-specific files should be in `platforms/{platform_name}/` subdirectories
- User guides go in `user_guide/` subdirectory
- Technical documentation goes in `technical/` subdirectory

### Content Organization
- Start with overview/summary section
- Include table of contents for longer documents
- Use consistent heading hierarchy (H1 ‚Üí H2 ‚Üí H3)
- Include code examples where relevant
- End with references or additional resources

### Markdown Standards
- Use descriptive link text for URLs
- Include alt text for images (when applicable)
- Use code blocks with language specification for syntax highlighting
- Use tables for structured data comparison
- Use emoji sparingly and consistently

## üîÑ Maintenance Guidelines

### When to Update Documentation
- After implementing new features or vulnerability detections
- When changing APIs or configuration options
- When fixing bugs that affect user-facing behavior
- When adding new platform support

### Documentation Review Process
1. Update relevant documentation files
2. Test examples and code snippets
3. Verify links and cross-references
4. Update table of contents if structure changes
